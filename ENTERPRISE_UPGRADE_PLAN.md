# План доработки проекта до уровня Enterprise Open-Source

В этом документе собран скорректированный план из 8 ключевых задач, которые позволят значительно повысить надежность, масштабируемость и прозрачность проекта `jira_timesheet`, не ломая при этом существующую архитектуру.

Задачи отсортированы в логическом порядке реализации: от настройки процессов разработки и стабильности интеграций до продвинутых метрик и Single Sign-On.

## 🗂️ Список задач (Roadmap)

- [ ] **1. CI/CD, сборка и мягкий линтинг**
  - Переход на **`uv`** как сверхбыстрый и современный менеджер зависимостей (взамен обычного `requirements.txt`).
  - Настройка **Ruff** с базовыми (мягкими) правилами для поддержания порядка в коде, без фанатизма (замена flake8/isort/black).
  - Создание простого бесплатного пайплайна в **GitHub Actions** (`.github/workflows/ci.yml`) для прогона тестов и линтера при каждом пуше. Без сложных self-hosted раннеров.

- [ ] **2. Асинхронные очереди (Background Workers) с Web UI**
  - Интеграция **SAQ (Simple Async Queue)** — современной, сверхбыстрой и полностью асинхронной очереди на базе Redis. Она идеально ложится на асинхронный стек FastAPI и предоставляет **отличную встроенную Web-панель** (Web UI) для управления и мониторинга задач.
  - Перенос тяжелой логики синхронизации с Jira из фоновых задач FastAPI в полноценные воркеры с надежной доставкой и мониторингом через UI.
  - Настройка стратегий повторных попыток (Retry с Exponential Backoff) на случай таймаутов API Jira.

- [x] **3. Интеграция по Webhooks и умный Поллинг**
  - Разработка эндпоинта (`POST /api/v1/sync/webhooks/jira`) для приема событий от Jira в реальном времени (`worklog_created/updated/deleted`).
  - Перевод текущего механизма поллинга (запросы с `since`) в режим нечастой фоновой сверки (например, раз в час-два).
  - Настройка сверки так, чтобы она проходила **последовательно** по каждому активному проекту, аккуратно размазывая нагрузку на сеть и БД в течение дня.

- [x] **4. Структурированное логирование (Structured JSON Logging)**
  - Внедрение **`structlog`** — это золотой стандарт Python для создания производительных и расширяемых JSON-логов, идеальный для FastAPI.
  - Перевод вывода логов в формат JSON (для удобного парсинга в Loki/Datadog) и добавление поля `trace_id` из мидлвари в каждую строчку лога для сквозной трассировки.

- [ ] **5. Интеграция Rate Limiting и продвинутого кэша в Redis**
  - Замена текущего in-memory словаря на распределенный Rate Limiter (например, `fastapi-limiter`), работающий через Redis.
  - Декорирование тяжелых запросов (особенно Report Builder) через `@cache` (`FastAPICache`).
  - **Критично:** Настройка адекватного TTL (времени жизни) для каждого кэша и явного **механизма инвалидации кэша** (сброс кэшей при ручной или webhook-синхронизации ворклогов).

- [ ] **6. Автоматический Audit Log (SQLAlchemy Events)**
  - Привязка миксинов и слушателей событий напрямую к моделям SQLAlchemy (`after_insert`, `after_update`, `after_delete`).
  - Автоматическое логирование изменений в критических таблицах с сохранением Diff'а («было» -> «стало» и кто изменил).

- [ ] **7. Метрики для Observability (Prometheus)**
  - Подключение `prometheus-fastapi-instrumentator`.
  - Добавление эндпоинта `/metrics` для мониторинга времени ответа запросов, количества 500-х ошибок, нагрузки на CPU/RAM.
  - **Версионирование:** Добавление метрики `build_info` (отслеживание текущей версии сборки), чтобы в Grafana было видно, после деплоя какой версии проекта поменялись графики (упала скорость или выросли ошибки).

- [ ] **8. Поддержка альтернативной аутентификации SSO (Authentik)**
  - Подключение библиотеки **`fastapi-sso`** (используя ее `Generic OIDC SSO` провайдер).
  - Настройка интеграции с корпоративным сервером аутентификации на базе **Authentik** по протоколу OpenID Connect (Без паролей).
  - Прозрачная авторизация или авто-регистрация пользователей, чей email от провайдера Authentik совпадает с локальной базой системы.
